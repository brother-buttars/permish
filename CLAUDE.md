# Permish — Development Guide

## Project Overview

A cross-platform app that digitizes the LDS Church "Parental or Guardian Permission and Medical Release Form." Event planners create pre-filled events with unique URLs; parents complete, sign, and submit forms. PDFs are generated by filling in the actual official church PDF. Supports web, desktop (Tauri), and mobile (Capacitor) with offline-first capability.

## Tech Stack

- **Frontend:** SvelteKit (Svelte 5 runes mode) + shadcn-svelte + Tailwind CSS v4
- **Backend (Express mode):** Node.js / Express 5 + SQLite (better-sqlite3)
- **Backend (PocketBase mode):** PocketBase + Node.js sidecar for PDF/email/SMS
- **Desktop:** Tauri v2 (PocketBase + sidecar as bundled sidecars)
- **Mobile:** Capacitor (iOS/Android) with @capacitor-community/sqlite
- **Offline:** sql.js (WebAssembly SQLite) with IndexedDB persistence
- **PDF:** pdf-lib (fills actual church PDF form fields)
- **Email:** Nodemailer (Gmail SMTP default, Resend-ready)
- **SMS:** Carrier email gateways via Nodemailer
- **Auth (Express):** JWT via HttpOnly SameSite=Strict cookies (bcryptjs)
- **Auth (PocketBase):** SvelteKit server-side cookie proxy preserving HttpOnly security
- **Package Manager:** pnpm (NOT npm)
- **Node Version:** 24 LTS (managed via nvm, see .nvmrc)
- **Deployment:** Docker Compose (Express or PocketBase profiles)

## Architecture

### Repository Pattern

The frontend uses a **repository pattern** that decouples all data access from any specific backend. Pages never import backend SDKs directly.

```
Pages/Components → getRepository() → DataRepository interface
                                        ├── Express adapter    (api.ts wrapper)
                                        ├── PocketBase adapter (PB SDK)
                                        ├── Local adapter      (sql.js SQLite)
                                        └── Hybrid adapter     (local + sync to PB)
```

**Key files:**
- `frontend/src/lib/data/types.ts` — domain types (User, Event, Submission, etc.)
- `frontend/src/lib/data/repository.ts` — interfaces (AuthRepository, EventRepository, etc.)
- `frontend/src/lib/data/adapters/express.ts` — Express backend adapter
- `frontend/src/lib/data/adapters/pocketbase.ts` — PocketBase adapter
- `frontend/src/lib/data/adapters/local.ts` — Offline-first local SQLite adapter
- `frontend/src/lib/data/sync/hybrid.ts` — Hybrid adapter (local + background sync)
- `frontend/src/lib/data/sync/manager.ts` — SyncManager (push/pull changes)
- `frontend/src/lib/data/index.ts` — Provider factory (`initRepository()`, `getRepository()`)

### Data Modes

Controlled by `localStorage` key `permish_data_mode`:

| Mode | Backend | Behavior |
|---|---|---|
| `online` (default) | Express or PocketBase | All data on server |
| `local` | sql.js (in-browser) | Fully offline, data never leaves device |
| `hybrid` | Local SQLite + PocketBase sync | Works offline, syncs when connected |

Backend selection via `PUBLIC_BACKEND` env var: `express` (default) or `pocketbase`.

### Backup/Restore

- `BackupManager` at `frontend/src/lib/data/backup/manager.ts`
- Exports local SQLite as `.permish-backup` file
- AES-256-GCM encryption with PBKDF2 key derivation from passphrase
- SHA-256 integrity checking on restore

## Development Commands

```bash
# Start backend (from backend/)
pnpm dev

# Start frontend (from frontend/)
pnpm dev                    # runs on port 3000

# Run backend tests
pnpm test                   # from backend/

# Build frontend
pnpm build                  # from frontend/

# Build for desktop
pnpm build:desktop          # from frontend/ (sets PocketBase env vars)
pnpm tauri:dev              # from frontend/ (requires Rust)
pnpm tauri:build            # from frontend/ (builds distributable)

# Mobile
pnpm mobile:ios             # build + sync + open Xcode
pnpm mobile:android         # build + sync + open Android Studio

# Docker Compose
docker compose up                              # Express mode (default)
PUBLIC_BACKEND=pocketbase docker compose --profile pocketbase up  # PocketBase mode

# Migration
node scripts/migrate-to-pocketbase.js          # SQLite → PocketBase
```

## Important Conventions

### Repository Pattern Usage

All pages use `getRepository()` from `$lib/data`:
```typescript
import { getRepository } from '$lib/data';
const repo = getRepository();
const events = await repo.events.list();
```

**Never import `$lib/api` directly** — it exists only as a reference. All data access goes through the repository.

### UI Components

- **Use shadcn-svelte components** for everything. If a component doesn't exist in shadcn-svelte, build it to match shadcn design patterns exactly.
- Components are hand-built in `frontend/src/lib/components/ui/` following exact shadcn patterns.
- All components use **Svelte 5 runes** (`$props()`, `$state()`, `$derived`, `$effect`, `$bindable()`).
- The `cn()` utility at `$lib/utils.ts` merges Tailwind classes.
- Theme colors use CSS variables defined in `app.css` (oklch format with dark mode).

#### Atomic design

The frontend follows atomic design: **atoms** (`components/ui/`, `components/atoms/`) → **molecules** (`components/molecules/`) → **organisms** (`components/organisms/`) → **composables** (`components/composables/`) → **pages** (`routes/**/+page.svelte`). The 3× rule: if a pattern repeats inline in three places, extract it to a molecule before the fourth. Pages orchestrate; they don't render headers, list cards, status pills, or filter bars inline. Before writing any new component, check `btrs/knowledge/conventions/registry.md` — it lists what exists and what's planned. Full rules: `btrs/knowledge/conventions/ui.md`. Anti-patterns: `btrs/knowledge/conventions/anti-patterns.md`.

### Field Name Consistency

Domain types in `frontend/src/lib/data/types.ts` are the single source of truth. Field names match the SQLite schema exactly:

| Correct Name | NOT This |
|---|---|
| participant_name | first_name, last_name |
| participant_dob | date_of_birth |
| participant_phone | phone (alone) |
| emergency_contact | emergency_contact_name |
| emergency_phone_primary | emergency_contact_phone |
| allergies_details | allergy_details |
| can_self_administer_meds | can_self_administer |
| event_description | description (alone) |
| event_name | name (alone for events) |
| event_dates | dates (alone) |
| submitted_at | created_at (for submissions) |

### Date Formatting

- All displayed dates use `formatDate()` from `$lib/utils/formatDate.ts`
- Date-only values: "20 March 2026"
- Datetime values (with timestamp): "20 March 2026 at 6:29 pm"
- Date INPUT fields keep YYYY-MM-DD format (for `<input type="date">`)

### Notifications

- Use toast system (`toastSuccess`, `toastError` from `$lib/stores/toast`)
- Never use `alert()` — always use toasts
- Toasts pop in from top center with auto-dismiss

### Modals / Confirmations

- Use `ConfirmModal` component (`$lib/components/ConfirmModal.svelte`) for all destructive/confirmation actions
- Blurred backdrop (`backdrop-blur-sm`)
- Never use `confirm()` — always use ConfirmModal

### Navigation

- Nav items have active states (primary color + bottom border)
- Events link is active for any `/event/*` subpage
- Both desktop and mobile nav must be updated together

### Organizations

- Young Men: Deacons, Teachers, Priests
- Young Women: Builders of Faith, Messengers of Hope, Gatherers of Light (DB keys remain `beehives`, `mia_maids`, `laurels`)
- Stored as JSON array on events table
- Display uses `getOrgDisplayLabels()` — shows parent label when all children selected
- Color badges: blue (Young Men family), purple (Young Women family)

### Auth

**Express mode:**
- JWT in HttpOnly SameSite=Strict cookies
- `extractUser` middleware runs on every request
- `requireAuth` / `requirePlanner` for protected routes

**PocketBase mode:**
- SvelteKit `hooks.server.ts` proxies PocketBase auth tokens into HttpOnly cookies
- API routes at `frontend/src/routes/api/auth/` manage cookie lifecycle
- PocketBase API rules enforce row-level security

**Local/Hybrid mode:**
- SHA-256 password hashing via Web Crypto API
- Session stored in localStorage (device-level security)

### PDF Generation

- Uses pdf-lib to fill in the actual official church PDF form fields
- Template PDF at `backend/src/templates/permission-form.pdf` (also in `sidecar/src/templates/`)
- Drawn signatures embedded as PNG images at field positions
- Typed signatures set as text in form fields
- Form is flattened after filling (fields become static)
- In PocketBase mode, PDF generation runs in the Node.js sidecar
- In local-only mode, PDF generation is not available (no server)

### Database

**Express mode:** SQLite via better-sqlite3, schema at `backend/src/db/schema.js`
**PocketBase mode:** PocketBase internal SQLite, schema at `pb_migrations/`
**Local mode:** sql.js (WebAssembly), schema at `frontend/src/lib/data/local/schema.ts`

### Testing

- Backend uses Jest + supertest
- Tests use in-memory SQLite (`:memory:`)
- `app.locals.db` is overridden in tests
- Rate limiters skip in test env (`NODE_ENV=test`)
- `crypto.randomUUID()` used instead of `uuid` package (ESM-only issue)

## Route Mounting Order (Critical — Express mode)

In `backend/src/index.js`, form routes MUST be mounted BEFORE events routes since both use `/api/events` prefix:

1. Auth routes (`/api/auth`)
2. Form routes (`/api/events` — public: `/:id/form`, `/:id/submit`)
3. Events routes (`/api/events` — planner-only, has `requireAuth + requirePlanner`)
4. Profiles routes (`/api/profiles`)
5. Submissions routes (`/api/submissions` — `/mine` BEFORE `/:id/*`)
