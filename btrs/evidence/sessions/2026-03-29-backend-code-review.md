# Backend Code Review Report

**Date:** 2026-03-29
**Scope:** All backend route handlers, middleware, services, config, and entry point
**Reviewer:** API Engineer Agent

---

## Summary

The backend is well-structured for a small Express 5 app. Route mounting order is correct, auth middleware is properly applied, and the separation between public form routes and planner-only event routes is sound. However, there are several issues ranging from security concerns to missing validation and error handling gaps.

**Totals:** 5 HIGH, 10 MEDIUM, 7 LOW

---

## HIGH Severity Issues

### H1. No global error handler -- unhandled async errors will crash Express 5

**File:** `backend/src/index.js`
**Lines:** entire file (missing)

**Description:** There is no Express error-handling middleware (`app.use((err, req, res, next) => {...})`). In Express 5, rejected promises from async route handlers are forwarded to the error handler, but without one defined the default Express handler sends an HTML stack trace in development and a bare 500 in production. More critically, synchronous throws in non-async handlers or errors from multer's `fileFilter` callback will go unhandled.

The multer upload in `events.js` line 32 does `cb(new Error('File type not allowed'))` -- without a global error handler this produces an HTML error response instead of JSON.

**Recommended fix:** Add a global error handler at the end of `index.js`:
```js
app.use((err, req, res, next) => {
  console.error('Unhandled error:', err);
  const status = err.status || 500;
  res.status(status).json({ error: err.message || 'Internal server error' });
});
```

### H2. JWT cookie maxAge is hardcoded, diverges from JWT token expiry

**File:** `backend/src/middleware/auth.js`, lines 38-45
**File:** `backend/src/config.js`, line 6

**Description:** The JWT token is signed with `config.jwtExpiry` (default `'24h'`), but the cookie `maxAge` is hardcoded to `24 * 60 * 60 * 1000` (exactly 24h). If someone changes `JWT_EXPIRY` to `'7d'`, the cookie will expire in 24h while the token is still valid for 7 days, or vice versa. These must be synchronized.

**Recommended fix:** Parse `config.jwtExpiry` into milliseconds and use it for both the JWT sign and cookie maxAge, or derive cookie maxAge from the same config value.

### H3. `uuid` package is listed as a dependency but should not be used

**File:** `backend/package.json`, line 38

**Description:** The `uuid` package is in `dependencies` but the codebase correctly uses `crypto.randomUUID()` everywhere. The convention explicitly says "crypto.randomUUID() instead of uuid package." This is dead weight that could confuse future contributors into using the wrong import.

**Recommended fix:** Remove `uuid` from `package.json` dependencies: `pnpm remove uuid`.

### H4. Path traversal risk in attachment download route

**File:** `backend/src/routes/form.js`, lines 42-53

**Description:** The attachment download route builds a file path from `attachment.filename` which comes from the database. While the filename is generated by multer using `crypto.randomUUID()` + extension, the `original_name` is used in the `Content-Disposition` header without sanitizing special characters. A malicious `original_name` containing characters like `"`, `\n`, or non-ASCII could lead to header injection.

Additionally, there is no validation that the resolved `filePath` is actually within `config.uploadsDir`. If a database record were ever tampered with, `path.join(config.uploadsDir, attachment.filename)` with a filename like `../../etc/passwd` would escape the uploads directory.

**Recommended fix:**
1. Sanitize `original_name` in the Content-Disposition header (or use the `content-disposition` npm package).
2. Add a path containment check:
```js
const filePath = path.join(config.uploadsDir, attachment.filename);
if (!filePath.startsWith(path.resolve(config.uploadsDir))) {
  return res.status(400).json({ error: 'Invalid filename' });
}
```

### H5. Form submission (`POST /:id/submit`) has no rate limiter applied

**File:** `backend/src/routes/form.js`, line 55
**File:** `backend/src/index.js`, line 42

**Description:** A `submitLimiter` is defined in `rateLimiter.js` (line 19-24) but is never used anywhere. The form submit endpoint is public (no auth required) and has no rate limiting at all. An attacker could flood the database with submissions, trigger PDF generation (CPU-intensive), and spam email/SMS notifications.

Similarly, `formLoadLimiter` is defined but never applied to the `GET /:id/form` route.

**Recommended fix:** Apply rate limiters in `index.js` or within `form.js`:
```js
// In form.js
const { submitLimiter, formLoadLimiter } = require('../middleware/rateLimiter');
router.get('/:id/form', formLoadLimiter, ...);
router.post('/:id/submit', submitLimiter, ...);
```

## MEDIUM Severity Issues

### M1. `PUT /api/auth/profile` has no input validation

**File:** `backend/src/routes/auth.js`, lines 64-79

**Description:** The profile update endpoint accepts any body content and writes it to the database. There is no validation for:
- `name` being non-empty (empty string would pass through `sanitizeString`)
- `phone` format
- `guardian_signature_type` being one of the valid enum values ('drawn', 'typed', 'hand')
- `guardian_signature` content (could be a massive string -- no size limit checked)

An attacker could set `guardian_signature_type` to any arbitrary string, bypassing the CHECK constraint (which would cause an unhandled SQLite error since there is no try/catch).

**Recommended fix:** Add validation middleware or inline checks before the UPDATE query, and wrap the DB call in try/catch.

---

### M2. Profiles routes (`POST /`, `PUT /:id`) have no required field validation

**File:** `backend/src/routes/profiles.js`, lines 15-41, 43-73

**Description:** Creating a child profile requires `participant_name` and `participant_dob` per the schema (NOT NULL), but the route handler does not validate these. If either is missing, the INSERT will throw a SQLite constraint error that is not caught, resulting in a 500 error with potentially leaked implementation details instead of a clean 400 response.

**Recommended fix:** Add explicit checks:
```js
if (!d.participant_name || !d.participant_dob) {
  return res.status(400).json({ error: 'Participant name and date of birth are required' });
}
```

---

### M3. `PUT /api/events/:id` allows setting any field to falsy value via `||` fallback

**File:** `backend/src/routes/events.js`, lines 107-120

**Description:** The update handler uses `sanitizeString(event_name || event.event_name)` pattern. This means if someone sends `event_name: ""` (empty string), the falsy check falls through to the old value, making it impossible to clear a field. More problematically, for fields like `leader_phone`, the `||` pattern means sending `"0"` would be treated as falsy in JS.

The `notify_email`, `notify_phone`, `notify_carrier` fields correctly use `!== undefined` checks, but the required fields use `||` which is inconsistent.

**Recommended fix:** Use explicit `!== undefined` or nullish coalescing (`??`) for all fields:
```js
sanitizeString(event_name !== undefined ? event_name : event.event_name)
```

### M4. `POST /api/events` does not validate or sanitize email fields

**File:** `backend/src/routes/events.js`, lines 44-58

**Description:** The `leader_email` and `notify_email` fields are accepted without email format validation. The `notify_phone` is accepted without phone format validation. `validateEmail` and `validatePhone` exist in `validate.js` but are not used in any event route.

**Recommended fix:** Apply `validateEmail` to `leader_email` and `notify_email`, and `validatePhone` to `leader_phone` and `notify_phone`.

---

### M5. `PUT /api/submissions/:id` does not validate required fields

**File:** `backend/src/routes/submissions.js`, lines 59-119

**Description:** The submission update endpoint does not validate that required fields (`participant_name`, `participant_dob`, `participant_signature_type`, `participant_signature_date`) are present in the request body. If any is `undefined`, the UPDATE will set the column to `NULL`, violating NOT NULL constraints and causing an unhandled SQLite error.

Also, `participant_age` uses `d.participant_age || submission.participant_age` -- if the DOB changed, the age should be recalculated, not taken from the request body or the old submission.

**Recommended fix:** Add the same validation as the submit route. Recalculate age from DOB.

### M6. `handlebars` package is unused

**File:** `backend/package.json`, line 28

**Description:** The `handlebars` template engine is listed as a dependency but is not imported or used anywhere in the codebase. All email templates use inline strings.

**Recommended fix:** Remove: `pnpm remove handlebars`.

### M7. Email notification sends PDF attachment without verifying the file exists

**File:** `backend/src/services/email.js`, lines 27-29

**Description:** The `sendNotification` function attaches a PDF by path without checking if the file actually exists or is readable. If PDF generation partially failed or the file was cleaned up, nodemailer would throw an error. While the caller catches this, the error message would be confusing.

**Recommended fix:** Check `fs.existsSync(pdfPath)` before adding the attachment, or handle the attachment conditionally.

### M8. Test setup does not run migrations

**File:** `backend/tests/setup.js`, lines 1-8

**Description:** The `createTestDb()` function calls `createTables(db)` but does NOT call `migrate(db)`. This means tests run against the base schema without migration columns like `organizations`, `event_start`, `event_end`, `youth_program`, and `additional_details`. Tests may pass coincidentally because SQLite is lenient about missing columns in some contexts, but tests cannot exercise features that depend on migrated columns.

**Recommended fix:** Add `migrate(db)` after `createTables(db)`:
```js
const { createTables, migrate } = require('../src/db/schema');
function createTestDb() {
  const db = new Database(':memory:');
  createTables(db);
  migrate(db);
  return db;
}
```

### M9. `events.js` `GET /:id/attachments` will shadow `form.js` `GET /:id/attachments` for planners

**File:** `backend/src/routes/events.js`, line 184
**File:** `backend/src/routes/form.js`, line 33

**Description:** Both `form.js` and `events.js` define `GET /:id/attachments`. Since form routes are mounted first in `index.js`, the form.js version handles all requests to this path. The events.js version (lines 184-191) is dead code -- it will never be reached because the form route always responds first (it does not call `next()`). The events.js version adds an ownership check (`created_by = ?`) and returns full attachment records with `SELECT *`, while the form.js version is public and returns limited fields. The planner-authenticated version never runs.

**Recommended fix:** Remove the duplicate route from `events.js` since it is unreachable. If planners need the full attachment records, handle that distinction in the form.js route or use a separate path.

### M10. `sanitizeString` silently passes non-string values through

**File:** `backend/src/middleware/validate.js`, lines 15-18

**Description:** `sanitizeString` returns the value unchanged if it is not a string: `if (typeof str !== 'string') return str;`. This means if a client sends `{ participant_name: 12345 }` (a number) or `{ participant_name: { "$gt": "" } }` (an object), it passes through unsanitized. While SQLite will coerce types, this defeats the purpose of sanitization.

**Recommended fix:** Return `null` or convert to string for non-string/non-null values:
```js
function sanitizeString(str, maxLength = 500) {
  if (str == null) return null;
  if (typeof str !== 'string') str = String(str);
  return str.slice(0, maxLength).replace(/[<>]/g, '');
}
```

## LOW Severity Issues

### L1. `require` inside route handler body

**File:** `backend/src/routes/auth.js`, line 66
**File:** `backend/src/routes/submissions.js`, line 72, 105-106

**Description:** `sanitizeString`, `generatePdf`, and `config` are required inside route handler functions rather than at the top of the file. While Node.js caches `require` calls so there is no performance issue, this is unconventional and makes dependency tracking harder.

**Recommended fix:** Move all `require` statements to the top of the file.

---

### L2. No `Content-Type` validation on `POST /:id/submit`

**File:** `backend/src/routes/form.js`, line 55

**Description:** The submit endpoint relies on `express.json()` middleware from `index.js`, but does not reject requests with wrong Content-Type. A `text/plain` request would result in `req.body` being `undefined`, causing a confusing `Cannot read property 'participant_name' of undefined` error.

**Recommended fix:** Add a check: `if (!req.body) return res.status(400).json({ error: 'Request body required' });`

---

### L3. `computeAge` function in form.js should handle invalid dates

**File:** `backend/src/routes/form.js`, lines 13-19

**Description:** If `participant_dob` is an invalid date string, `new Date(dob)` returns `Invalid Date`, and the age calculation produces `NaN`. This `NaN` is then inserted into the database as `participant_age`, which is defined as `INTEGER NOT NULL`.

**Recommended fix:** Validate the DOB format before computing age, return a 400 if invalid.

---

### L4. Event DELETE is soft-delete but orphaned submissions remain accessible

**File:** `backend/src/routes/events.js`, lines 126-132
**File:** `backend/src/routes/submissions.js`, lines 9-18

**Description:** Deleting an event sets `is_active = 0` but does not affect submissions. The `/submissions/mine` endpoint shows all submissions regardless of event active status. This is likely intentional (parents should still see their submitted forms), but there is no indicator to the parent that the event has been deactivated.

**Recommended fix:** Consider including `e.is_active` in the `/mine` query so the frontend can display deactivation status.

---

### L5. `config.js` uses `'dev-secret-change-me'` as default JWT secret

**File:** `backend/src/config.js`, line 5

**Description:** The default JWT secret is a known string. In development this is fine, but if someone deploys without setting `JWT_SECRET`, all tokens are signed with this public value. There is no warning or startup check.

**Recommended fix:** Add a startup check in `index.js` that warns (or exits in production) if `JWT_SECRET` is the default value:
```js
if (process.env.NODE_ENV === 'production' && config.jwtSecret === 'dev-secret-change-me') {
  console.error('FATAL: JWT_SECRET must be set in production');
  process.exit(1);
}
```

**File:** `backend/src/routes/form.js`, lines 66-71

**Description:** The submit endpoint checks for presence of `participant_signature_date` and `participant_dob` but does not validate they are valid date strings. ## Additional Observations (not bugs, but worth noting)

1. **Express 5 compatibility**: The codebase correctly uses Express 5 (`^5.2.1`). Express 5 auto-forwards rejected promises from async handlers, so the `async` handlers in `auth.js` and `form.js` are properly handled. However, synchronous throws in the multer `fileFilter` callback still need the global error handler (covered in H1).

2. **Route mounting order**: Confirmed correct -- `formRoutes` before `eventsRoutes` on `/api/events`, and `/mine` before `/:id` in submissions. This is sound.

3. **Database access pattern**: Using `req.app.locals.db` consistently across all routes, with test override via `app.locals.db = createTestDb()`. This is clean.

4. **CORS configuration**: Single-origin CORS with credentials is appropriate for this app.

5. **PDF service**: The pdf-lib usage is solid with proper error handling via try/catch around field operations. The `setText` and `setYesNo` helpers gracefully handle missing fields.
